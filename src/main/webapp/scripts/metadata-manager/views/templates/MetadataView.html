${moduleCode}, powered by Fireball

Use the Metadata Manager to build out your Fireball dashboard. This screen allows you to configure application-level elements including properties. You can then specify data sets in the "Data" tab and subsequently configure Menu Options, Filters and Views using the corresponding tabs.

Application Properties

Configure global application properties here. Use the button below to customize any of the following properties (as well as adding any app-specific properties):

  • title: The title for your dashboard. This shows up at the top and is the HTML document title.
  • help_url: The URL to your application's help content. If specified, a help icon (?) is shown on the top right of the dashboard.
  • logout_url: The URL a user should be redirected to upon logout. Defaults to /pkmslogout
  • premierconnect_url: The URL for the PremierConnect breadcrumb. Defaults to https://premierconnect.premierinc.com

    • + property

    Data Sets

    For many use cases you can define data sets here, and a RESTful web service will be automatically configured that will execute the query and map the data as specified. After you have defined data sets here, you can associate them with filters and other widgets on the subsequent tabs.

    + data set

    Menu Options

    Menu options are shown beneath the icon on the top right of the screen. Configure all menu options here and specify which are present on each view within that view.

    + menu option

    Filters

    Filters are displayed across the top of the dashboard. Filters can remain across views or show/hide as needed. Configure all navigation filters here and specify which are present for each view within that view.

    + filter

    Multi-Screen Header Widgets

    The same widget can often appear in the header across multiple screens. Rather than having to create multiple instances of that widget and populate it with the same data, just define a "header" widget here.

    + widget

    Advanced

    Use this screen to modify the metadata directly. Be careful, as changes here are automatically persisted when the text area is blurred (assuming it is valid JSON). Invalid JSON is not persisted until corrected. Note that property names must be enclosed in strings. Also keep in mind that referential integrity can be compromised if you choose to do so!

    Frequently Asked Questions

    The Fireball documentation on CODE is the best source of current information.


    What is Fireball?

    You should probably have a good idea if you've come this far, but here's the overview: https://code.premierinc.com/docs/display/FIREBALL/Fireball

    I need a new feature added. Who does that?

    It depends. The Fireball team does have a backlog and will happily incorporate new functionality in the framework. However, if you don't have the time to wait or would just prefer to add it yourself, you have a few options. If what you need is not something anyone else could ever benefit from, you can always just write that custom functionality in your dashboard's codebase. Otherwise, we greatly appreciate anyone who is willing to branch Fireball, make your changes, and submit a pull request for review. See https://code.premierinc.com/docs/display/FIREBALL/Fireball for more info on that.

    Where do I submit bugs/issues?

    Here

    I've changed some metadata. Now how do I backup those changes and push them to the project?

    The changes have already been made to the "${moduleCode}.json" file in your project's WEB-INF/metadata directory. Just commit and push!

    Why is the metadata represented with JSON?

    Because. The UI loves JSON, so why over-complicate things with a database dependency and corresponding convoluted table structure. JSON is incredibly fast and flexible. It also allows us to sandbox metadata changes within an environment rather than immediately distributing them to all users of a shared data source prior to the rest of the codebase being in sync.
